## Функциональные требования: Расчёт допуска и получения зачёта

### 1. Цель
*Автоматизировать расчёт итогового результата («зачёт» / «незачёт») на основе множес*тва критериев, включая успеваемость, посещаемость, активность и академическую историю.

### 2. Исходные данные (входные параметры)

Система должна принимать следующие данные по каждому студенту за текущий семестр:

| Параметр | Тип | Диапазон | Описание |
|----------|-----|----------|-----------|
| Количество сданных лабораторных работ | целое | 0..N (общее число ЛР) | Принимаются только полностью и правильно выполненные работы |
| Посещаемость (%) | вещественное | 0–100 | Процент присутствия на занятиях |
| Выполненные тесты (%) | вещественное | 0–100 | Средний процент правильных ответов по всем тестам |
| Активность на занятии | целое/балл | 0–10 | Оценка, выставляемая преподавателем (или автоматически, по числу ответов) |
| Написание конспектов лекций | булево / % | 0 или 100% | Зачёт/незачёт по факту сдачи всех конспектов (или процент сданных) |
| Академическая задолженность за прошлый семестр | булево | да / нет | Есть ли долги по другим дисциплинам на момент расчёта |

### 3. Правила расчёта (бизнес-логика)

#### 3.1. Блок обязательных условий (блокирующие факторы)

Зачёт **НЕ СТАВИТСЯ** автоматически, если выполняется хотя бы одно из условий:

1. **Академическая задолженность за прошлый семестр** = «Да» → незачёт (без права сдачи зачёта в текущем семестре, либо только после ликвидации долгов — настраивается).
2. **Посещаемость** < минимального порога (например, 50%) → незачёт.
3. **Лабораторные работы** — сдано менее порогового значения (например, < 60% от общего числа ЛР).
4. **Конспекты лекций** — не сданы в полном объёме (если параметр бинарный: false → незачёт).

#### 3.2. Взвешенная оценка (для остальных случаев)

Если все обязательные условия выполнены, система вычисляет **итоговый рейтинговый балл** (0–100) по формуле:

```
Итоговый балл = 
    K_лаб * (сданные ЛР / общее число ЛР * 100)
  + K_пос * (посещаемость)
  + K_тест * (средний % тестов)
  + K_акт * (активность × 10)   // переводим 0–10 в 0–100
  + K_конс * (% сданных конспектов)
```

где **K_лаб + K_пос + K_тест + K_акт + K_конс = 1** (коэффициенты настраиваются, например: 0.3, 0.2, 0.2, 0.1, 0.2).

#### 3.3. Порог получения зачёта

- Если **Итоговый балл >= Порог_зачёта** (например, 60 баллов) → «Зачёт».
- Иначе → «Незачёт».

### 4. Особые случаи

1. **Автоматический зачёт** (опционально): если студент набрал >= 90 баллов по итогам, зачёт ставится автоматически без учёта блокирующих факторов (кроме академической задолженности).
2. **Индивидуальные условия**: преподаватель может вручную скорректировать итог (повысить/понизить) с обязательным комментарием.
3. **Академическая задолженность** — если долг ликвидирован до начала сессии, система позволяет пересчитать зачёт (должен быть механизм переоткрытия расчёта).

### 5. Результат расчёта (выходные данные)

Для каждого студента система формирует:

- Итоговый статус: **«Зачёт»** / **«Незачёт»**.
- Детализированный отчёт:
  - Значения по каждому критерию.
  - Итоговый балл.
  - Причина блокировки (если есть).
  - Дата расчёта.
  - Кто выполнил расчёт (автоматически / вручную преподаватель).

### 6. Требования к интерфейсу

- Возможность массового расчёта по группе.
- Возможность индивидуального пересчёта.
- Ручная корректировка коэффициентов преподавателем (на уровне дисциплины).
- Экспорт результатов в Excel/PDF.

### 7. Нефункциональные требования (кратко)

- Расчёт всей группы (до 50 чел.) — не более 2 секунд.
- Журналирование всех изменений (кто, когда, что менял).
- Разграничение ролей: студент (видит только свой результат), преподаватель (расчёт, корректировка), администратор (настройка весов и порогов).
